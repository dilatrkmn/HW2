# Production Deployment Recommendation

For production, run the crawler and API/UI as separate processes sharing a durable datastore. Keep the API tier stateless behind a load balancer and place crawl workers in an autoscaled worker group pulling tasks from a persistent queue (e.g., Redis streams, RabbitMQ, or cloud queue). Move from SQLite to PostgreSQL (or a dedicated search engine such as OpenSearch) to improve concurrent write/read throughput and support richer ranking. Add robust URL canonicalization, robots.txt compliance, domain allow/deny policies, request retries with exponential backoff, and content-type/language filtering to improve crawl quality and legal/safety posture.

To support search during active indexing at scale, use incremental indexing with near-real-time refresh. A practical approach is dual-write: persist canonical page records in OLTP storage and push tokenized updates into a search index via a streaming pipeline. Maintain crawl job checkpoints, queue offsets, and idempotent task keys so interruptions can resume cleanly. Instrument the system with metrics (queue lag, fetch latency, success/error rates, index freshness), tracing, and SLO alerts. This architecture preserves the same product surface while making it resilient for high-volume continuous crawling.
